# AGENTS.md

اقرأ هذا الملف أولًا قبل أي تعديل في المشروع.

## اقرأ قبل البدء
1. اقرأ `.codex/skills/SKILL.md`
2. إذا وُجد، اقرأ `.codex/work-progress.json`
3. أكمل من آخر `last_stable_checkpoint` أو من آخر مرحلة مكتملة

## المهمة
طوّر **المشروع الحالي نفسه** فقط، ولا تنشئ مشروعًا جديدًا من الصفر.

## لا تكسر هذه الأشياء
- الشكل العام الحالي للموقع
- شكل القائمة الرئيسية الحالي
- التوزيع الحالي للصفحات
- اللغة العربية كلغة أساسية
- تنظيم JSON الحالي بحسب التصنيف
- الأمثلة والمشاريع الكاملة الموجودة إلا إذا كان الاستبدال ضروريًا فعلًا

## المكتبات المشمولة
- C++
- CMake
- GLSL
- Vulkan
- Dear ImGui
- SDL3
- SDL3_image
- SDL3_mixer
- SDL3_ttf

## الأولويات
1. تعديل المشروع الحالي فقط
2. الحفاظ على الشكل الحالي
3. الحفاظ على القائمة الرئيسية الحالية
4. الحفاظ على JSON بحسب التصنيف
5. إعادة استخدام وتحسين الأمثلة والمشاريع الكاملة الموجودة
6. تحسين البحث ليكون exact-match first ثم الأقرب ثم الجزئي
7. تحسين Tooltip للمبتدئ
8. شرح المعنى البرمجي الحقيقي بالعربية
9. تحسين الأمثلة والتعليقات العربية
10. تحسين الأداء وJavaScript والتحميل الذكي
11. الروابط الداخلية
12. المرجع الرسمي
13. الأيقونات وSVG عند الحاجة فقط

## سير العمل الإلزامي
1. افحص بنية المشروع الحالية
2. اختر **مرحلة صغيرة واحدة فقط**
3. قبل التعديل المهم، أنشئ نقطة رجوع داخل `.codex/checkpoints/`
4. نفّذ أقل تغيير يحقق الفائدة
5. اختبر التعديل
6. حدّث `.codex/work-progress.json`
7. انتقل للمرحلة التالية

## ملف التتبع
يجب إنشاء أو تحديث:
- `.codex/work-progress.json`

## نقاط الرجوع
احفظ نقاط الرجوع داخل:
- `.codex/checkpoints/`

## قواعد JSON
- حافظ على JSON بحسب التصنيف
- لا تخلط `functions` مع `structs` أو `examples` عشوائيًا
- أضف الحقول الجديدة تدريجيًا
- حافظ على التوافق الرجعي إن أمكن

## قواعد البحث
- الأولوية للتطابق التام
- ثم التطابق القريب / prefix
- ثم التطابق الجزئي
- ثم المرادفات والمفاهيم المرتبطة
- اجعل الفهارس خفيفة
- لا تحمّل فهرسًا كبيرًا كاملًا في أول زيارة
- قسّم الفهارس بحسب المكتبة أو التصنيف عند الحاجة

## قواعد JavaScript والأداء
- اجعل JavaScript خفيفًا وصغيرًا
- حمّل فقط ما تحتاجه الصفحة الحالية
- استخدم lazy loading / dynamic loading عند الحاجة
- لا تضع كل البيانات في ملف واحد كبير
- لا تحمّل كل tooltips أو examples أو search data دفعة واحدة
- راعِ الإنترنت البطيء والأجهزة الضعيفة

## قواعد Tooltip
اجعله مختصرًا وواضحًا للمبتدئ، ويشمل عند الحاجة:
- الاسم
- النوع
- المكتبة
- المعنى البرمجي الحقيقي
- الهدف
- ماذا يفعل فعليًا
- خطأ شائع
- رابط الصفحة

## قواعد الأمثلة
- استخدم الأمثلة والمشاريع الموجودة أولًا
- حسّنها بدل استبدالها متى أمكن
- أضف مثالًا سريعًا ومثالًا كاملًا عند الحاجة
- أضف تعليقات عربية على السطور المهمة
- أضف تلوين كود وزر نسخ
- أضف روابط داخلية وTooltip للرموز المعروفة
- لا تحمّل ملفات الأمثلة الكبيرة إلا عند الحاجة

## المرجع الرسمي
أضف في نهاية صفحة العنصر، عند الحاجة:
- رابط المرجع الرسمي
- اسم المصدر
- روابط مرتبطة مفيدة

## أسلوب الشرح
اكتب بالعربية التقنية المبسطة للمبتدئ:
- لا تستخدم ترجمة حرفية فقط
- اشرح المعنى البرمجي الحقيقي
- اشرح الهدف
- اشرح ماذا يفعل فعليًا
- اشرح ما لا يفعله عند الحاجة لتجنب اللبس

## تقليل استهلاك التوكن
- لا تكتب شروحات طويلة بلا حاجة
- لا تكرر ما تم إنجازه سابقًا
- اعتمد على `.codex/work-progress.json`
- اجعل التقارير قصيرة جدًا

## صيغة التقرير بعد كل مرحلة
- `phase`
- `done`
- `changed`
- `tested`
- `next`
- `issues`
- `context_updated`

## القاعدة الذهبية
حسّن ما هو موجود، ولا تعِد بناء المشروع من الصفر.
