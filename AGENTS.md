2. إذا وُجد، اقرأ `.codex/work-progress.json`

## المهمة
طوّر **المشروع الحالي نفسه فقط**، ولا تنشئ مشروعًا جديدًا من الصفر، ولا تغيّر هوية المشروع أو بنيته العامة إلا إذا كان ذلك ضروريًا جدًا ومبررًا بوضوح.

## لا تكسر هذه الأشياء
- الشكل العام الحالي للموقع
- شكل القائمة الرئيسية الحالي
- التوزيع الحالي للصفحات
- اللغة العربية كلغة أساسية
- تنظيم JSON الحالي بحسب التصنيف
- الأمثلة والمشاريع الكاملة الموجودة، إلا إذا كان الاستبدال ضروريًا فعلًا
- الروابط والمسارات الحالية ما أمكن
- أسلوب العرض الحالي ما لم يكن التحسين موضعيًا وواضح الفائدة

## قواعد التفسير والشرح
- اشرح العناصر شرحًا برمجيًا حقيقيًا، وليس وصفًا سطحيًا أو إنشائيًا.
- عند شرح دالة أو متغير أو بنية أو كلمة محجوزة أو نوع أو مفهوم، وضّح:
  - ما هو
  - ماذا يعني داخل اللغة أو النظام البرمجي
  - لماذا وُجد
  - ماذا يفعل فعليًا
  - ما الذي لا يفعله عند الحاجة لتجنب اللبس
  - متى يُستخدم
  - متى لا يُستخدم
- لا تكتب تعريفات عامة بلا معنى فعلي.
- لا تكتفِ بعبارات مثل: "يخزن قيمة" أو "ينفذ أمرًا" دون توضيح المعنى البرمجي الحقيقي.
- اجعل الشرح مناسبًا للمبتدئ، لكن دقيقًا وغير مخل.
- استخدم العربية التقنية المبسطة، لا الترجمة الحرفية الجامدة.
- عند وجود مفهوم يلتبس على المبتدئ، وضّحه بمثال وفكرة ذهنية مختصرة.

## قواعد المرجع الرسمي
- اعتمد المرجع الرسمي أو المرجع الأساسي للمفهوم كلما كان ذلك مناسبًا.
- في صفحات C++، اعتمد خصوصًا على:
  - `https://en.cppreference.com/`
  - `https://isocpp.github.io/CppCoreGuidelines/CppCoreGuidelines`
- في المواضع ذات الصلة بلغة C، يمكن الاعتماد على:
  - `https://cppreference.com/w/c/index.html`
- ابنِ الشرح على المعنى الحقيقي الوارد في المرجع، ثم أعد صياغته عربيًا بشكل واضح للمبتدئ.
- لا تضع ادعاءات غير مدعومة بمعنى المرجع.
- أضف في نهاية صفحة العنصر، عند الحاجة:
  - رابط المرجع الرسمي
  - اسم المصدر
  - روابط مرتبطة مفيدة

## قواعد خاصة بشرح C++
- عند شرح مفاهيم اللغة، تعامل مع C++ ككيان مستقل.
- لا تجعل شرح مفاهيم اللغة تابعًا لمكتبات أو أطر خارجية.
- لا تستخدم أمثلة من مكتبات خارجية إلا إذا كانت الصفحة نفسها عن التكامل معها.
- ركّز على المعنى اللغوي الحقيقي لعناصر مثل:
  - variables
  - functions
  - parameters
  - return values
  - references
  - pointers
  - const
  - scope
  - lifetime
  - structs
  - classes
  - constructors
  - destructors
  - keywords
  - expressions
  - statements
  - types

## قواعد JSON
- حافظ على JSON بحسب التصنيف
- لا تخلط `functions` مع `structs` أو `examples` عشوائيًا
- أضف الحقول الجديدة تدريجيًا
- حافظ على التوافق الرجعي إن أمكن
- لا تُدخل حقولًا جديدة عامة على كل المشروع دفعة واحدة دون حاجة
- ابدأ بالحقول الأكثر فائدة والأقل كسرًا للبنية الحالية

## قواعد البحث
- الأولوية للتطابق التام
- ثم التطابق القريب / prefix
- ثم التطابق الجزئي
- ثم المرادفات والمفاهيم المرتبطة
- اجعل الفهارس خفيفة
- لا تحمّل فهرسًا كبيرًا كاملًا في أول زيارة
- قسّم الفهارس بحسب المكتبة أو التصنيف عند الحاجة
- فضّل تحسين الدقة أولًا قبل توسيع الذكاء الدلالي

## قواعد JavaScript والأداء
- اجعل JavaScript خفيفًا وصغيرًا
- حمّل فقط ما تحتاجه الصفحة الحالية
- استخدم lazy loading / dynamic loading عند الحاجة
- لا تضع كل البيانات في ملف واحد كبير
- لا تحمّل كل tooltips أو examples أو search data دفعة واحدة
- راعِ الإنترنت البطيء والأجهزة الضعيفة
- تجنّب الأكواد العامة الثقيلة إذا كان الحل الموضعي يكفي

## قواعد Tooltip
اجعله مختصرًا وواضحًا للمبتدئ، ويشمل عند الحاجة:
- الاسم
- النوع
- المكتبة أو التصنيف
- المعنى البرمجي الحقيقي
- الهدف
- ماذا يفعل فعليًا
- خطأ شائع
- رابط الصفحة

قواعد إضافية:
- لا تكرر عنوان العنصر بصياغة مرادفة فقط
- اجعل الـ Tooltip مفيدًا فعلًا لا مجرد تلميح شكلي
- لا تجعله طويلًا لدرجة يزعج القراءة
- أضف رابطًا داخليًا عند وجود صفحة مرتبطة مفيدة

## قواعد الروابط الداخلية
- أضف روابط داخلية فقط عندما تخدم الفهم فعلًا
- اربط بين المفاهيم المتقاربة مثل:
  - المتغير ↔ النوع
  - الدالة ↔ المعامل ↔ القيمة المرجعة
  - النطاق ↔ العمر الزمني
  - المرجع ↔ المؤشر
  - struct ↔ class
  - constructor ↔ object lifetime
- تجنب الإكثار من الروابط بلا فائدة

## قواعد الأمثلة
- استخدم الأمثلة والمشاريع الموجودة أولًا
- حسّنها بدل استبدالها متى أمكن
- أضف مثالًا سريعًا ومثالًا كاملًا عند الحاجة
- أضف تعليقات عربية على السطور المهمة
- أضف تلوين كود وزر نسخ
- أضف روابط داخلية وTooltip للرموز المعروفة عند الحاجة
- لا تحمّل ملفات الأمثلة الكبيرة إلا عند الحاجة
- اشرح السطور المهمة بعد المثال
- اجعل الأمثلة توضح المعنى، لا الصياغة فقط
- عند الحاجة، استخدم:
  - مثالًا بسيطًا
  - مثالًا أوضح في الاستخدام
  - مثالًا لخطأ شائع

## مرجع نهج التصميم الحالي للإضافات الجديدة
اعتمد هذا الملخص كخط أساس عند إضافة صفحات أو أقسام أو عناصر جديدة، ولا تخرج عنه إلا لسبب واضح:

- استخدم القوالب الموحدة الحالية من عائلة `parameter-detail-card` و`content-card` و`prose-card` بدل اختراع واجهة جديدة لكل مكتبة.
- حافظ على ألوان المشروع الحالية والخلفيات الداكنة الهادئة ولمسات `accent` الحالية. لا تدخل خلفيات بيضاء أو ألوانًا غريبة عن الهوية الحالية.
- في صفحات C++ وغيرها من الصفحات المرجعية الجديدة:
  - ابدأ ببطاقة معلومات أساسية واضحة
  - ثم معنى حقيقي واستخدام فعلي
  - ثم أمثلة متعددة
  - ثم عناصر مرتبطة
  - ثم المرجع الرسمي
- إذا احتاج العنصر إلى ملف تضمين، فأظهره بوضوح كسطر فعلي مثل `#include <iostream>` داخل بطاقة بارزة.
- إذا كان الاسم يحتاج نطاقًا مثل `std::`, فأظهر طريقة كتابته الصحيحة، وعند الحاجة وضّح بديل `using std::name;` وتجنب جعل `using namespace std;` هو المسار الافتراضي.
- الأمثلة الجديدة يجب أن تكون متعددة لا أقل من ثلاثة متى كان ذلك منطقيًا:
  - مثال بسيط
  - مثال أوضح في الاستخدام
  - مثال أذكى أو أقرب لخطأ شائع أو حالة عملية
- أمثلة الكود يجب أن تستخدم نفس shell الكود الموجود في المشروع مع زر نسخ وزر طي/إظهار، لا قالب كود جديد منفصل.
- بعد كل مثال، اشرح السطور أو الأجزاء المهمة بلغة عربية تقنية واضحة، مع التركيز على سبب السطر لا إعادة قراءته فقط.
- `عناصر مرتبطة` تُعرض ككتلة واحدة متماسكة داخل بطاقة واحدة، مع روابط داخلية وtooltip، لا كسرد مفكك ولا كنص خام.
- أقسام مثل `الدوال والخصائص المهمة` أو ما يشابهها تُعرض طوليًا بطاقة تحت أخرى إذا كان المحتوى تحليليًا، لا كشبكة مضغوطة.
- عناوين الحقول داخل البطاقات التحليلية مثل `ما الذي تفعله؟` و`متى تفيد؟` و`الفائدة الفعلية` و`الكلفة أو السرعة` يجب أن تكون مميزة بصريًا ضمن نفس ألوان المشروع.
- عند شرح عمليات الأنواع أو الدوال، اذكر فائدتها الفعلية وما أمكن كلفتها أو سرعتها أو أثرها العملي، لا الاسم وحده.
- اشرح المعنى البرمجي الحقيقي بالعربية، وتجنب الحشو والترجمة الحرفية والمصطلحات الإنجليزية غير الضرورية في النصوص التفسيرية.
- اجعل C++ واللغات الأخرى كيانات مستقلة في الشرح، ولا تربط المعنى بمكتبات خارجية إلا إذا كانت الصفحة عن التكامل أصلًا.
- كل إضافة جديدة يجب أن تحافظ على:
  - الروابط الداخلية
  - tooltip المفيد
  - المرجع الرسمي في نهاية الصفحة
  - التدرج البصري نفسه
  - خفة JavaScript وعدم تحميل بيانات غير لازمة
